Open Design's workflow is four steps. Let me walk through them in order.

Step one, pick a starting point. Describe what you want in one sentence, or pick a starting point straight from templates and Plugins — today's deck, for example, started from a presenter-mode Slides Plugin.

Step two, set the visual direction. Once the direction is locked in, the palette, fonts, and spacing automatically carry into the rest of the generation flow — no per-slide hand-tuning.

Step three, generate the artifact — the most important step. The agent reads the full context and produces real, runnable files — not screenshots, not preview images — that you can preview and edit instantly in the sandbox.

Step four, deliver and build memory. The files can go straight to engineering for further development, or become a marketing video via HyperFrames; meanwhile every choice you make is captured as memory, so the next generation lands closer to what you want.

So what's the philosophy behind this workflow? Next part.

// part 03 · principles

Open Design's three principles

01
🖥️ Desktop-native

Design happens on the desktop. Local files, Figma exports, and code repos are directly readable, and the agent has full terminal execution power.

02
🤖 Don't build agents — plug into them

The Claude Code / Codex / Cursor on your machine are already powerful. Open Design wires them into a complete design workflow.

03
🧠 Gets smarter with every use

Every choice settles into your Design System, preferences, and memory — so the next generation lands closer to what you want.

Now for the heart of today's talk — Open Design's three principles.

First, desktop-native. The desktop is where design actually happens: local files, Figma exports, code repositories — all directly readable. More importantly, the agent here has full terminal execution power — something no cloud tool can give you.

Second, don't build agents — plug into them. We don't reinvent the wheel. The Claude Code, Codex, and Cursor on your machine are already strong; what Open Design does is wire them into a complete design workflow — 21 coding agents supported so far, auto-detected from your PATH, zero configuration.

Third, it gets smarter the more you use it. Every choice settles into your Design System, preferences, and memory, so the next generation lands closer to what you want. That's a self-evolving design system.

Put the three together: open, local, agent-native. Finally, let's see how to get started.
