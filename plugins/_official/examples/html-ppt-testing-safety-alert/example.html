Three flows, three risk profiles

L1 · Green

Local-first

Open Design: designs, memory, and SQLite all live in your local .od/; BYOK connects straight to the provider.
Risk: you're in control

L2 · Amber

Partly in the cloud

Cloud SaaS design tools: files live with the vendor, export formats are limited, and a shutdown cuts you off.
Risk: at the vendor's mercy

L3 · Red

Fully in the cloud

Closed-source cloud AI design: prompts, artifacts, and keys all pass through a third party, model locked to one vendor.
Risk: unauditable

Your API key never leaves your machine

In BYOK mode, keys live only in your local .od/media-config.json, and requests go straight to the model provider — no intermediary servers.

# .od/ · Open Design local data directory · everything stays on your machine
local_data:
  - app.sqlite          # sessions, memory, preferences
  - projects/<id>/      # agent working directories
  - artifacts/          # saved design artifacts
  - media-config.json   # API keys · local only

byok_direct:
  - providers: [anthropic, openai, azure, google]
    route: your machine → provider, no servers in between

never_uploaded:
  - "your prompts"
  - "your design files"
  - "your API keys"

tool-audit checklist06 / 08

7 questions to ask before you choose a tool

✓
Where do design files live? Can you git diff them? (Open Design: local filesystem)
✓
Does your API key pass through third-party servers? (BYOK: stored locally, direct to provider)
!
Cloud tools: will your prompts and artifacts be used for training? Most terms won't say.
✓
If you go offline or the vendor shuts down, can you still open past work? (Local files: always)
!
Cloud tools: model locked to one vendor — no agent swapping, no self-hosting, no BYOK.
✓
Can you audit the code line by line? (Apache-2.0, fully open — fork, self-host, commercial use all allowed)
!
Cloud tools: can you export your preference and memory data — and truly delete it?
